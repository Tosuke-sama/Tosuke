React原理剖析（一）

发表于2024-01-17|更新于2024-01-17|学习前端框架

React

forwardRef

使用forwardRef来包裹函数组件，使得函数组件拥有额外的ref参数，将此ref参数传递进dom标签的ref中即可做到转发的目的。

useImperativeHandle

在被forwardRef所包围的组件中，向父组件的ref转发的不仅可以是dom的ref，你还可以转发一些自己的函数操作，这样可以间接的使得父组件调用dom上的方法而不必直接转发暴露dom。

其中的第二个参数为可选的 dependencies，也就如同effect一样，在可选的 dependencies改变时，useImperativeHandle将会重新执行。

render

render的作用是根据一次更新中产生的新状态值，通过 React.createElement ，替换成新的状态，得到新的 React element 对象，新的 element 对象上，保存了最新状态值。 createElement 会产生一个全新的props。到此 render 函数使命完成了。

渲染控制

  1. shouldComponentUpdate(nextProps, nextState, context) 控制是否进行render
  2. PureComponent会将props state浅比较，当发生变化时才会重新render 当同时存在PureComponent和shouldComponentUpdate同时存在时，shouldComponentUpdate 的权重，会大于 PureComponent
  3. React.memo(Component, isEqual)：可以对比 props 变化，来决定是否渲染组件。第二个参数是一个对比函数，参数是新旧props，如果返回true表示相等，则需要重新render，未提供的情况下会将props进行浅比较
  4. useMemo(fn, deps)：用于缓存一个值，仅当依赖发生变化时，才会重新执行函数获取新的返回值
  5. useCallback(fn, deps)：用于缓存一个函数

reducer

对于大量的state修改任务，将其放置于reducer中。

  1. 将任务修改为使用dispatch函数（参数为此次修改任务的信息）
  1. 编写一个reducer函数
  1. 将state升级为reducer

dispatch就成为了修改tasks的方法，这种加强的方法当然也需要在创建时添加额外的方法：也就是tasksReducer。

Fiber的起源

是虚拟DOM的进化版，在React15之前，reconciler会递归创建虚拟DOM，递归过程无法中断，这就会造成卡顿。React16将递归的无法中断的更新重构为异步的可中断更新，由于曾经用于递归的虚拟DOM数据结构已经无法满足需要。于是，全新的Fiber架构应运而生。

Fiber的含义：

  1. 数据的保存由递归的调用栈转为了Fiber节点
  2. 每个Fiber节点对应着一个React element，保存了该组件的类型，对应的DOM节点等信息。
  3. Fiber节点保存了更新中要更改的状态与要执行的工作。

JSX

JSX在编译时会被Babel编译为React.createElement方法。
JSX是一种描述当前组件内容的数据结构，他不包含组件schedule、reconcile、render所需的相关信息。
例如什么优先级啊，reconciler打上的标记啊等等。
因为这些内容都包含在Fiber节点中。
在mount时，reconciler根据jsx所描述的内容生成对应的fiber节点，在update的时候根据jsx的信息来为fiber节点打上标记

构建fiber树

render阶段开始于performSyncWorkOnRoot或performConcurrentWorkOnRoot，其中一个是同步更新另一个是异步更新

performUnitOfWork用来创建fiber节点并且连接至workInProgress。

performUnitOfWork的递归工作

首先从rootFiber开始向下深度优先遍历。为遍历到的每个Fiber节点调用beginWork方法。

该方法会根据传入的Fiber节点创建子Fiber节点，并将这两个Fiber节点连接起来。
在“归”阶段会调用completeWork (opens new window)处理Fiber节点。

beginWork

beginWork的工作是传入当前Fiber节点，创建子Fiber节点。

current：当前组件对应的Fiber节点在上一次更新时的Fiber节点，即workInProgress.alternate,在第一次渲染的时候由于不存在上一次的节点，所以说current是null。
workInProgress：当前组件对应的Fiber节点。
renderLanes：优先级相关。

renconcileChildren

  • 对于mount的组件，他会创建新的子Fiber节点

  • 对于update的组件，他会将当前组件与该组件在上次更新时对应的Fiber节点比较（也就是俗称的Diff算法），将比较的结果生成新Fiber节点

  • export function reconcileChildren(
  current: Fiber | null,
  workInProgress: Fiber,
  nextChildren: any,
  renderLanes: Lanes
) {
  if (current === null) {
    // 对于mount的组件
    workInProgress.child = mountChildFibers(
      workInProgress,
      null,
      nextChildren,
      renderLanes,
    );
  } else {
    // 对于update的组件，该方法会添加effectTag标签至fiber
    workInProgress.child = reconcileChildFibers(
      workInProgress,
      current.child,
      nextChildren,
      renderLanes,
    );
  }
}

在updateHostComponent内部，被处理完的props会被赋值给workInProgress.updateQueue，并最终会在commit阶段被渲染在页面上。

mount时

mount时的主要逻辑包括三个：

  • 为Fiber节点生成对应的DOM节点
  • 将子孙DOM节点插入刚生成的DOM节点中
  • 与update逻辑中的updateHostComponent类似的处理props的过程

在之前的effect tag中我们提到mount时只有root节点才具有tag，为了减少dom操作，其实就是因为递归时在appendAllChildren中我们已经拼接好了一颗离屏dom树。

effectList

在以上过程中，所有的存在effecttag的fiber节点将会保存至一条被叫做effectList的链表中，也就是递归的过程中。

至此，render阶段全部工作完成。在performSyncWorkOnRoot函数中fiberRootNode被传递给commitRoot方法，开启commit阶段工作流程。
